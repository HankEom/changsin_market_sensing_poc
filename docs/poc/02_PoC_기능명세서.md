## 2. PF-01: 트렌드 데이터 수집

### 기능 설명

글로벌 패키징 미디어 3개 이상 소스에서 화장품 용기 관련 콘텐츠를 크롤링하여 Supabase에 저장한다.

### 처리 흐름

```
사용자 [수집 실행] 클릭 또는 cron 트리거
  → Next.js API Route (/api/crawl)
  → 소스별 크롤러 병렬 실행
  → HTML 파싱 → 정제 → 중복 체크 (URL 기준)
  → Supabase collected_articles 테이블 INSERT
  → 수집 결과 요약 반환 (신규 N건 / 중복 M건 / 실패 K건)
```

### 입출력

| 항목 | 내용 |
|------|------|
| 트리거 | POST /api/crawl (수동) 또는 Vercel Cron |
| 입력 | 없음 (수집 대상은 코드에 정의) |
| 출력 | `{ total: number, new: number, duplicate: number, failed: number }` |
| 저장 | collected_articles 테이블 |

### 예외 처리

| 상황 | 처리 |
|------|------|
| 사이트 응답 없음 | 10초 타임아웃 → 해당 소스 스킵 → 로그 기록 |
| 파싱 실패 | 해당 기사 스킵 → 로그 기록 |
| Supabase 연결 실패 | 에러 반환 → 프론트에서 재시도 안내 |

## 3. PF-02: 키워드 추출 및 트렌드 지수 생성

### 기능 설명

수집된 기사에서 Claude API를 사용하여 화장품 용기 관련 키워드(형태·소재·마감)를 추출하고, 키워드별 트렌드 지수를 계산한다.

### 처리 흐름

```
POST /api/analyze 호출
  → Supabase에서 미분석 기사 조회 (analyzed = false)
  → 기사 N건씩 배치로 Claude API 호출
    → 키워드 추출 (형태/소재/마감 분류 포함)
  → 기사별 추출 키워드 → article_keywords 테이블 저장
  → 기사 analyzed = true 업데이트
  → 키워드별 주간 빈도 집계 → 전주 대비 변화율 계산
  → trend_keywords 테이블 UPSERT
  → 급증 키워드 (변화율 > 50%) 탐지 → 알림 대상 플래그
```

### 트렌드 지수 계산 공식

```
trend_index = mention_count × (1 + change_rate / 100)

- mention_count: 최근 7일 언급 횟수
- change_rate: 전주 대비 변화율 (%)
- 급증 판정: change_rate > 50%
```

### 입출력

| 항목 | 내용 |
|------|------|
| 트리거 | POST /api/analyze |
| 입력 | 없음 (미분석 기사 자동 조회) |
| 출력 | `{ analyzed: number, keywords_extracted: number, surges: string[] }` |

## 4. PF-03: 트렌드-용기 LLM 매칭

### 기능 설명

트렌드 키워드와 용기 카탈로그를 Claude API에 전달하여 적합도를 평가하고 매칭한다. (pgvector 임베딩 파이프라인 제거 — Claude 텍스트 매칭으로 단순화)

### 처리 흐름

```
POST /api/match 호출
  → trend_keywords에서 Top 5 또는 급증 키워드 조회
  → container_catalog에서 활성 용기 전체 조회
  → Claude API 호출: "키워드와 각 용기의 적합도를 0~100으로 평가"
    → 키워드당 Top 3 매칭 (적합도 ≥ 60)
  → 매칭 결과 → match_results 테이블 저장
  → 결과 반환
```

### 입출력

| 항목 | 내용 |
|------|------|
| 트리거 | POST /api/match |
| 입력 | `{ keyword?: string }` (선택, 없으면 Top 5 전체) |
| 출력 | `MatchResult[]` (키워드명, 용기코드, 적합도 점수, 적합 이유, 용기 상세) |

## 5. PF-04: 선제 제안서 자동 생성

### 기능 설명

매칭 결과를 기반으로 Claude API가 영업팀용 선제 제안서를 **일반 POST 응답**으로 생성한다. (~~SSE 스트리밍 제거~~ — 로딩 스피너 + 완성 텍스트 표시로 변경)

### 처리 흐름

```
POST /api/proposal 호출
  → 매칭 결과 + 트렌드 분석 데이터 조합
  → Claude API 일반 호출 (제안서 프롬프트)
  → 완성된 텍스트 반환 (Markdown)
  → proposals 테이블에 저장
```

### 제안서 구조

| 섹션 | 내용 |
|------|------|
| 트렌드 요약 | 키워드명, 급증률, 시장 동향 근거 |
| 추천 용기 3종 | 용기코드, 스펙, 적합도 점수, 제안 이유 |
| 예상 효과 | 트렌드 선행 제안의 기대 효과 |
| 창신 강점 | 해당 트렌드 대응 역량 |

### 입출력

| 항목 | 내용 |
|------|------|
| 트리거 | POST /api/proposal |
| 입력 | `{ keyword: string, containers: string[] }` |
| 출력 | `{ success: true, data: { content: string (Markdown), id: string } }` |

## 8. PF-10: 수집 수동 트리거

### 기능 설명

대시보드에서 [수집 실행] 버튼 클릭 시 전체 파이프라인(수집→분석→매칭)을 순차 실행한다.

### 실행 흐름

```
[수집 실행] 클릭
  → Step 1: /api/crawl (크롤링) — 프로그레스 표시
  → Step 2: /api/analyze (분석) — 프로그레스 표시
  → Step 3: /api/match (매칭) — 프로그레스 표시
  → 완료: 결과 요약 토스트 표시 + 대시보드 갱신
```

### UI 상태

| 상태 | 표시 |
|------|------|
| 대기 | [수집 실행] 버튼 활성 |
| 실행 중 | 스피너 + 현재 단계 표시 ("크롤링 중..." → "분석 중..." → "매칭 중...") |
| 완료 | 성공 토스트 + 결과 수치 |
| 실패 | 에러 토스트 + 재시도 버튼 |
